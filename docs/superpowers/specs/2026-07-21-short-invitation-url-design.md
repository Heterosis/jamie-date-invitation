# Short Invitation URL Design

Date: 2026-07-21

Status: Approved in collaborative design review

Deployment target: GitHub Pages

## 1. Summary

Add a compact, opaque, self-contained invitation URL while preserving the existing static deployment and legacy query-string contract. New links generated by the maker use this shape:

```text
https://heterosis.github.io/jamie-date-invitation/s/#<payload>
```

The payload contains the complete invitation configuration in a versioned, compressed, Base64URL-encoded envelope. It requires no API, database, redirect service, cookies, local storage, or server-side code.

Existing root query URLs remain valid and unchanged:

```text
https://heterosis.github.io/jamie-date-invitation/?to=Jamie&from=Alex&...
```

## 2. Goals

- Make newly generated invitation links substantially shorter and less visually revealing.
- Keep the application completely static on GitHub Pages.
- Keep each short invitation self-contained so it never depends on a lookup service.
- Preserve all existing invitation values, fallbacks, sanitization, Calendar behavior, and Telegram behavior.
- Preserve every existing root query URL without redirects or migration.
- Keep codec versions evolvable without exposing a visible `v1.` prefix.
- Make malformed or unsupported short links fail clearly and safely.

## 3. Non-goals

- No backend, API, database, edge function, redirect service, or third-party URL shortener.
- No expiry, revocation, click analytics, account system, or access control.
- No encryption or claim that an opaque payload is secret.
- No human-readable or hand-editable short payload.
- No dynamic `/s/<payload>` pathname. Arbitrary path segments cannot reliably return HTTP 200 from the current GitHub Pages deployment.
- No automatic rewriting of existing query URLs into short URLs.

## 4. Current constraints

The application is a Vite and TypeScript static site with one runtime entry point. It currently reads `location.search`, opens maker mode for `?make=1`, and otherwise renders an invitation. The maker writes every non-empty form value into a self-contained query URL. GitHub Actions uploads only the built `dist` artifact to GitHub Pages.

The current parser is the security and validation boundary. It cleans text, enforces length limits, validates date, time, time zone, duration, and Telegram usernames, and supplies deterministic fallbacks. The short-link decoder must reconstruct input for this parser rather than render decoded values directly.

## 5. Public URL contract

### 5.1 Short invitation

```text
<site-base>/s/#<payload>
```

- `/s/` selects short invitation mode.
- The fragment contains exactly one unpadded Base64URL token.
- Query parameters on `/s/` do not select maker mode and are ignored by the short-link decoder.
- An empty fragment is an invalid short invitation.
- The page does not redirect after decoding, so the address bar and subsequent shares retain the short URL.

### 5.2 Legacy invitation and maker

- `<site-base>/?...` continues to use the existing query parser.
- `<site-base>/?make=1` continues to open maker mode.
- Legacy links keep their existing fallback and error behavior.
- Short-link routing takes precedence whenever the normalized pathname is `<site-base>/s/`.

## 6. Build and routing architecture

The repository adds a source `s/index.html` that imports the same application entry point as the root document. Vite's `build.rollupOptions.input` explicitly names both HTML entries; relying on the development server's SPA fallback is not sufficient. The production build therefore emits both:

- `dist/index.html`
- `dist/s/index.html`

Both documents load the same application entry point and shared assets. The short entry is a real static file, so a direct request to `<site-base>/s/` returns HTTP 200 without a `404.html` SPA fallback.

Path detection and URL construction use the configured Vite base path. They do not hard-code the GitHub owner, repository name, or production origin. Local development, root-domain deployments, renamed repositories, and the current GitHub Pages project path therefore use the same logic.

No client-side router is introduced.

## 7. Encoding data flow

The maker follows this sequence:

1. Read and normalize `MakerValues` with the existing defaults.
2. Validate date, time, time zone, and duration through the existing production parser and event-window validation.
3. Narrow the parsed result to a shareable invitation with non-null date and time and `make: false`.
4. Convert that shareable invitation into a canonical sparse tuple.
5. Serialize the tuple as UTF-8 JSON.
6. Compress the bytes with raw DEFLATE.
7. Prefix the compressed bytes with the binary codec-version byte.
8. Encode the complete envelope as unpadded Base64URL.
9. Build `<site-base>/s/#<payload>` with the standard `URL` API.

The maker uses that same short URL for the generated field, iframe preview, clipboard action, and Web Share action. Previewing therefore exercises the same route and decoder that the recipient will use.

## 8. Compact tuple schema

The codec accepts this narrower domain type rather than every possible application state:

```ts
type ShareableInvitationConfig = Omit<InvitationConfig, "date" | "time" | "make"> & {
  readonly date: string;
  readonly time: string;
  readonly make: false;
};
```

The uncompressed JSON value has this form:

```text
[mask, ...presentValues]
```

`mask` is an unsigned 12-bit integer. Values follow in ascending bit order. A set bit means that the corresponding value is present.

| Bit | Field | Serialized representation | Omitted when |
| ---: | --- | --- | --- |
| 0 | `to` | String | Value is `Jamie` |
| 1 | `from` | String | Empty |
| 2 | `date` | Eight-digit `YYYYMMDD` integer | Never in a maker-generated short link |
| 3 | `time` | Integer minutes after midnight, `0` through `1439` | Never in a maker-generated short link |
| 4 | `tz` | IANA time-zone string | Value is `Asia/Singapore` |
| 5 | `duration` | Integer minutes | Value is `120` |
| 6 | `place` | String | Empty |
| 7 | `title` | String | Equal to the parser's title derived from `from` |
| 8 | `note` | String | Equal to `DEFAULT_NOTE` |
| 9 | `telegram` | String without a leading `@` | Missing |
| 10 | `notifyName` | String | Equal to `from`, or `ME` when `from` is empty |
| 11 | `tgText` | String | Missing |

The generator requires valid date and time, so bits 2 and 3 are always set in newly generated tokens. The decoder rejects tokens that omit either bit. The date decoder left-pads the integer to eight digits before reconstructing strict `YYYY-MM-DD`; the time decoder reconstructs strict `HH:mm`.

`make` is never serialized. Unknown mask bits, extra tuple values, missing tuple values, wrong types, non-integer numeric fields, and out-of-range numeric fields are invalid.

## 9. Binary envelope and compression

Version 1 uses this envelope:

```text
[0x01][raw-DEFLATE UTF-8 JSON tuple]
```

The entire envelope is encoded with the RFC 4648 URL-safe alphabet:

- `+` becomes `-`.
- `/` becomes `_`.
- Trailing `=` padding is removed.

The visible token therefore matches `^[A-Za-z0-9_-]+$`. The version is recoverable from the first decoded byte but is not displayed as a textual prefix.

`fflate` is added as an exact-pinned runtime dependency. Only its synchronous raw-DEFLATE helpers are imported so Vite can tree-shake unrelated formats and streaming utilities. Standard `TextEncoder` and a fatal-mode `TextDecoder` handle UTF-8 so malformed byte sequences are rejected instead of replaced. Version 1 uses compression level 9; payloads are small enough that this synchronous work does not affect interaction responsiveness.

Future formats receive a new envelope byte. An implementation must never reinterpret an unknown version as version 1.

## 10. Decoding data flow

Short invitation mode follows this sequence:

1. Read the fragment without the leading `#`.
2. Validate the token alphabet and encoded-length limit.
3. Decode Base64URL into bytes.
4. Read and validate the envelope version.
5. Inflate the remaining raw-DEFLATE bytes with a strict output limit.
6. Decode UTF-8 and parse JSON.
7. Validate the tuple shape, mask, value count, types, numeric ranges, and required date/time bits.
8. Reconstruct long-form parameter names and values in `URLSearchParams`.
9. Pass the reconstructed search string to `parseInvitationConfig`.
10. Render the normal invitation from the resulting frozen `InvitationConfig`.

Decoded content is never inserted into the DOM before it passes through the existing parser and view boundaries.

## 11. Limits and malformed links

The decoder enforces both limits below:

- Encoded fragment: at most 8,192 characters.
- Inflated UTF-8 JSON: at most 8,192 bytes.

Inflation uses `inflateSync` with a caller-provided 8,193-byte output buffer. A result that fills the sentinel byte is rejected, so valid output can contain at most 8,192 bytes and the decoder never performs an unbounded output allocation. `fflate` can still process more compressed instructions before returning a truncated buffer; the 8,192-character encoded-input cap bounds that residual CPU risk but does not claim constant-time decoding.

Any of the following produces the invalid-invitation view:

- Missing or empty fragment.
- Characters outside the Base64URL alphabet, including padding.
- Invalid Base64URL length or byte sequence.
- Missing or unknown version byte.
- Empty, truncated, corrupt, or oversized DEFLATE data.
- Invalid UTF-8 or JSON.
- Invalid tuple, mask, field type, numeric range, or date/time presence.
- Any unexpected codec exception.

The application does not fall back to the default invitation because that could make a corrupt personalized link look valid.

## 12. Invalid-invitation experience

The invalid state uses the existing visual language and renders a clear heading and this recipient-facing guidance:

```text
This invitation link couldn't be opened.
Please ask the sender for a new link.
```

It does not echo the payload, expose exception details, show the maker entry, or offer a path that could be mistaken for accepting or declining an invitation. The document title and heading clearly identify the error, and the static message remains readable without animation.

## 13. Maker and integration behavior

- Maker mode remains at the root `?make=1` URL.
- The generated output becomes the short URL only.
- Copy and Web Share keep their current fallback and cancellation behavior.
- The iframe previews the short URL, not a hidden long equivalent.
- Google Calendar continues to receive validated event fields and never receives the invitation payload.
- Telegram continues to use the invitation page's `location.href`. Because the short page does not redirect, generated Telegram drafts contain the short URL exactly once.
- Reset and maker validation behavior remain unchanged.

## 14. Module boundaries

The implementation introduces small modules with single responsibilities:

- `payload-schema` converts between a validated invitation and the compact tuple.
- `payload-codec` owns the version envelope, UTF-8 conversion, raw DEFLATE, Base64URL, and size limits.
- `short-url` recognizes short mode and constructs the public short URL using the configured base.
- `invalid-invitation-view` renders only the malformed-link state.

The existing parser continues to own sanitization, validation, and fallbacks. Rendering modules do not decode payloads, and codec modules do not touch the DOM.

## 15. Compatibility and migration

- All existing query URL fixtures and public links remain valid.
- The root parser contract and documented field names remain supported.
- No redirect, canonicalization request, or network lookup occurs for legacy links.
- Newly generated links use short format immediately after deployment.
- A future codec may decode older envelope versions, but version 1 behavior remains immutable once released.

## 16. Privacy and security

The fragment is not sent in the HTTP request to GitHub Pages. It is still visible to the recipient, browser history, clipboard, screenshots, browser extensions with sufficient access, and any messaging service through which the complete URL is shared.

Compression and Base64URL are encoding, not encryption. Anyone who obtains the link can decode its invitation data. The README must continue to warn users not to include secrets.

The feature adds no analytics, tracking pixels, cookies, browser storage, credentials, or external data service. Size limits and strict schema validation reduce denial-of-service and malformed-input risks. All decoded text continues through the existing length and control-character sanitation before rendering or external-link construction.

## 17. Testing strategy

### 17.1 Unit tests

- Round-trip every field in `ShareableInvitationConfig`; `make` and incomplete date/time states are intentionally outside the codec domain.
- Round-trip every permitted default, derived, and empty optional value, plus Unicode, emoji, and maximum-length text.
- Verify default and derived fields are omitted from the tuple.
- Verify dates and times round-trip through their compact numeric forms.
- Verify identical canonical configurations produce identical tokens.
- Verify tokens contain only the unpadded Base64URL alphabet.
- Reject empty, malformed, padded, truncated, corrupt, unknown-version, invalid-UTF-8, oversized, and decompression-limit inputs.
- Reject unknown mask bits, missing/extra values, wrong types, invalid ranges, and missing required date/time bits.
- Verify decoded data still receives all `parseInvitationConfig` sanitation and fallbacks.
- Preserve all existing query parser and maker URL tests.

### 17.2 Browser tests

- Maker produces a URL whose pathname ends in `/s/` and whose fragment contains one opaque token.
- The generated URL does not expose long field names or invitation text.
- The short URL iframe renders the exact invitation values.
- Copy and Web Share receive the short URL.
- A fresh direct navigation to `/s/#<payload>` renders the invitation and returns HTTP 200 from the static short entry.
- Missing and malformed payloads render the accessible invalid-invitation state without console errors.
- YES followed by Telegram produces a draft containing the short URL exactly once.
- Calendar behavior remains unchanged.
- Legacy query URLs continue to render and behave identically.
- A production build contains both `dist/index.html` and `dist/s/index.html`.
- Browser verification against `vite preview` loads the built `/s/` document and all assets with HTTP 200.
- The deployed-site smoke test covers both a legacy root query invitation and a short `/s/#<payload>` invitation, including the main-document status and repository-base asset paths.

### 17.3 Reference length fixture

Use this canonical invitation for the URL-length regression test:

| Field | Value |
| --- | --- |
| `to` | `Jamie` |
| `from` | `Alex` |
| `date` | `2026-08-08` |
| `time` | `19:30` |
| `tz` | `Asia/Singapore` |
| `duration` | `120` |
| `place` | `Botanic Gardens` |
| `title` | Parser-derived default |
| `note` | `DEFAULT_NOTE` |
| `telegram` | `alex_date` |
| `notifyName` | `Alex` |
| `tgText` | Missing |

At the current production origin and repository path, the complete short URL must be no longer than 140 characters and must be shorter than the equivalent legacy URL.

The canonical tuple for this fixture is exactly:

```json
[590,"Alex",20260808,1170,"Botanic Gardens","alex_date"]
```

The encoded payload may contain at most 87 characters, leaving room for the current 53-character production `/s/#` prefix. An independent raw-DEFLATE design probe produced a 79-character payload and a 132-character complete URL. The exact-pinned `fflate` implementation must satisfy the 87-character payload gate; tests do not require a byte-for-byte compressor snapshot because raw DEFLATE remains interoperable across compliant implementations.

### 17.4 Build gate

The feature is releasable only when unit tests, browser tests, type checking, the production build, and the deployed-site smoke test pass.

## 18. Acceptance criteria

1. Maker-generated links use `<site-base>/s/#<payload>` and require no network lookup.
2. Direct short-link navigation returns a real static page rather than relying on a 404 fallback.
3. Every field in a complete, non-maker `ShareableInvitationConfig` round-trips through the version 1 codec and existing parser.
4. Existing root query links remain fully compatible.
5. Malformed short links fail closed with the dedicated recipient-facing error state.
6. Telegram re-sharing preserves the short URL and includes it exactly once.
7. The reference fixture meets the 140-character limit.
8. The app remains free of backend services, persistence, analytics, credentials, and runtime network dependencies beyond its existing user-triggered integrations.
9. README documentation explains both URL formats, the opaque-not-encrypted privacy model, and the malformed-link behavior.
10. The complete verification suite and deployed smoke test pass.
